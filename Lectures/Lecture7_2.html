Implementing the breadth first search:

We implement the breadth first search according to the above strategy of generating all possible path by one step extension and then ordering these paths in order to expand the next one, etc.

/* (breadthF PATHS SOL) holds if SOL is the path from the start to the goal node when the current candidate path from the collection of paths PATH has been expanded.
Initially PATHS contains only one path candidate, the list with only one node, the START node.

This means that the top level call to breadthF is

?((breadthF ((START)) SOL))

*/

The main clause for breadthF will then do the following:

  • generate all expansions  by one step of the current path, that is,  the head of the list PATHS.
  • add the result (a list of partial paths) at the end of PATHS.
  • invoke breadthF on the new PATHS to obtain the solution.
Therefore we have the following definition

((breadthF (PAth | TPaths) SOL)
                             (expand PAth  NPaths)
                             (append TPaths NPaths NEWPATHS)
                             (breadthF NEWPATHS SOL))
 

The  stopping rule for this condition is then obtained when the current path to be expanded starts with the goal node, that is:

((breadthF ((X |T) |  PATHS) (X|T))
                           (goal X))

The predicate expand is similar to solve_all_wfa from which we will remove the predicates operating on the knowledge base and replace go by path.  That is, from the predicate

((solve_all_wfa S G SOLUTIONS)
                             (kill goal)
                             (addcl ((goal G)) )
                             (findall P ((go S P (S))  SOLUTIONS)
                             (delcl ((goal G)) ))

we will obtain:

((expand (HPath|TPath) NPaths)
                               (findall  (NEXT HPath|TPath)
                                                ((path HPath NEXT)(negg member NEXT (HPath|TPath)))
                                                NPaths
                                )
)

We next wrap this in a top level predicate  solve_BF as follows:

((solve_BF S  G SOL)
                                    (kill goal)
                                    (addcl ((goal G)))
                                    (breadthF ((S)) S1)
                                    (reverse S1 SOL)
                                    (pretty_print_solution SOL)
                                    (delcl  ((goal G))  ))
 

Note that breadth first builds the path in reverse order, and therefore I used reverse to output the path in direct order, from the start to the goal.

The efficiency of FRIL (in general logic programming) programs is not the issue here. There are many ways in which the breadth fisrt search program above can be improved, including:

  • implementing the behavior of findall (rather than using the bip version)
  • adding/deleting  clauses in the knowledge base, rather than producing the list of partial paths : this is very dangerous, but it has the potential of being much faster since the fastest operation for FRIL is to look up the knowledge base.
Real world search  problems  may exceed in complexity even very clever implementations of breadth first search.  In these cases new algorithms, which make use of some additional information regarding the algorithm's performance are used.
 
 

Heuristic search

Associate a cost with each path: Best first search

Greedy :
   Minimize the estimated cost of the cheapest path from the current state to the goal state.

h(n) = extimated cost of the cheapest path from n to the goal state.

A* :
   Minimize the total path cost.

f(n) = total path cost of the cheapest solution from the node n to the goal.
g(n) = cost so far (can be computed exactly).
h(n) = extimated cost of the cheapest path from n to the goal state.

f(n) = g(n) +  h(n)

f: evaluation function.
h : heuristic function.
g: cost function.

Properties required of h:

admissibility: h must never overestimate the real cost of reaching the goal : h*. (example of h: bird flight, or straight line distance).


Properties for f:
 

If h is admissible then f(n) never overestimates the cost of the cheapest path: f*.

Monotonicity: along any path f-cost never decreases.  To realize this property we use the maxpath equation:


f(n') = max (f(n) , g(n') + h(n'))

 

where n is a parent of n'.

Properties of A*:

  • A* is optimal: it finds the higest quality solution.
  • A* is complete: it is guarranteed to find a solution.
  • Complexity:
    •   computation: exponential growth unless the error in the heuristic function grows at most as fast as  the logarithm of the actual path cost:
    | h(n) -h*(n)|   less or equal O(log h*(n))
    • heavy use of space.
